# ADR-0029: `phai serve install --system` — root LaunchDaemon on port 80 via one admin-auth prompt

- Status: active (extends ADR-0028)
- Date: 2026-06-15

## Context

ADR-0028 made `phai serve install` write a launchd **agent** in the user's
`gui/<uid>` domain. That agent cannot bind privileged ports (< 1024): on the
default port 80 it fails with `Permission denied` and crash-loops under
`KeepAlive`. Port 80 is what makes the friendly host `http://phai.localhost/`
work — without it the family-member install lands on `http://localhost:N/`,
which is exactly the kind of detail that audience never remembers.

The only way to bind port 80 on macOS is a **root LaunchDaemon** in the
`system` domain. Until now the only path to that was a hand-typed sequence of
`sudo` commands (write the plist, `chown root:wheel`, `chmod 644`,
`launchctl bootstrap system …`). That works but is exactly the "open a terminal
and run these commands" experience ADR-0028 set out to eliminate, and it is
error-prone (one wrong path and the daemon silently never loads).

The BigQuery backend authenticates via an absolute `service_account_path` in
`config.toml`, so a root process authenticates fine — provided `HOME` and the
captured `PHAI_CONFIG_DIR` are present in its environment (a root daemon
inherits none of the user's environment).

## Decision

**`phai serve install --system` installs the root LaunchDaemon itself, running
every privileged step through a single native macOS admin-auth prompt
(`osascript -e 'do shell script "…" with administrator privileges'`) instead of
a manual `sudo` sequence.** `phai serve uninstall --system` removes it the same
way. Without `--system`, the user-agent behavior from ADR-0028 is unchanged.

Mechanics:

- The daemon plist (`/Library/LaunchDaemons/run.phai.serve.plist`, Label
  `run.phai.serve`, RunAtLoad + KeepAlive, default port 80) is generated by a
  pure builder, captures `HOME` plus the usual `PHAI_CONFIG_DIR`/`FINANCE_OS_*`
  env, and runs with `--no-auto-update` so the root process never rewrites its
  own binary. It logs to the machine-wide `/Library/Logs/phai/serve.log`.
- The plist body is staged to a temp file first; the privileged shell command
  only `cp`s it into place, fixes ownership/mode, boots out any prior instance,
  and bootstraps the new one. The command is built by single-quoting every
  path, so spaces are handled and **no env value can inject shell syntax**.
  That `/bin/sh` command is wrapped for AppleScript (escaping `\` and `"`) and
  handed to `osascript`, which pops the password dialog and runs it as root.
- The launcher `Phai.app` bundle (ADR-0028) is still written under `$HOME`
  without elevation, and the Dock behavior is unchanged.

## Options considered

- **Single admin-auth `osascript` prompt** (chosen): one native password
  dialog, no terminal, no `sudo` literacy required. Cons: the privileged
  command is assembled as a string (mitigated by single-quoting every
  interpolated path and unit-testing the builders); still installs an unsigned,
  user-writable binary as a root service (see below).
- **Print the manual `sudo` sequence for the user to run**: zero new code, but
  it is the exact terminal ceremony ADR-0028 exists to remove, and a mistyped
  path silently no-ops.
- **Signed/notarized `.pkg` installer**: the right long-term answer — installs
  into a root-owned location, plays nicely with Gatekeeper, and removes the
  user-writable-binary vector. Deferred: it needs an Apple Developer signing
  identity and notarization in the release pipeline, which is more than this
  change should pull in. This ADR is the bridge to that polish.

## Consequences

### Security trade-off (accepted)

The root daemon executes the per-user binary at `~/.local/bin/phai`, which is
writable by the (non-root) user. On a multi-user machine this is a **local
privilege-escalation vector**: anyone who can write that path gets code
execution as root at the daemon's next launch. We accept this on a
single-admin personal Mac because it preserves `phai self update` — the updater
rewrites that same user-writable path in place, so a root-owned binary would
break auto-update. The `--system` install prints a one-line warning naming the
path, and the module header + this ADR document the trade-off. A
signed/notarized `.pkg` installing into a root-owned location is the intended
future hardening that removes this vector.

### Other

- "Install phai on port 80 for a family member" is now a single friendly
  command with one password prompt, instead of a copied-and-pasted `sudo`
  block.
- `--system` is macOS-only (like the rest of the file) and idempotent
  (bootout-then-bootstrap; bundle rewritten in place). User cancellation of the
  auth dialog (`osascript` error -128) surfaces as a clear "authorization
  cancelled" error rather than a partial install.
- The plist/shell/osascript builders are pure functions with unit tests; the
  privileged tools (`osascript`, `launchctl`) are never invoked in tests.
- Port changes still require re-running install; the `run.phai.serve` label
  stays singleton across both the agent and daemon variants, so a machine
  should run one or the other, not both.
